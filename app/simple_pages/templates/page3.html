Workflows: A workflow is a configurable automated process that will run one or more jobs. Workflows are defined by a YAML file checked in to your repository

  • dev.yml: Development workflow which builds and tests pull requests.
  • prod.yml: Production workflow which builds and tests push requests.

.pytest_cache: offer a mechanism through which other plugins can set values via config. cache. Also stores test failures.

  • .gitignore: gitignore file tells Git which files to ignore when committing your project to the GitHub repository.
  • CACHEDIR.TAG: The cache directory is where files are stored during background uploads and after upload for quick access should you need to open the file from the network drive
  • README.md: A README file is an essential guide that gives other developers a detailed description of your GitHub project.

context_processors: Is a list of dotted Python paths to callables that are used to populate the context when a template is rendered with a request.

  • init_.py: Is a feature in flask called blueprint that acts as package of controllers such as plugins.
  • run.py: The runpy module is used to locate and run Python modules without importing them first.

simple_pages: The Simple Pages plugin allows you to easily create web pages for your public site using a simple form. My website includes one Index/Home page and an additional 4 pages which explain the technologies I learned in IS601 class so far.

Static: A Static Website (sometimes called a flat or stationary page) is displayed in a web browser exactly as it is stored. It contains web pages with fixed content coded in HTML and stored on a web server.

  • css: is a stylesheet language used to describe the presentation of a document written in HTML.
  • images: Images which I picked to use on my carousels of the index page.
  • js: JavaScript is a text-based programming language used both on the client-side and server-side that allows you to make web pages interactive.

Tests: contains a series of tests such as pytest that check the integrity of the app prior to deploying and merging.

.coverageerc: Code coverage is a metric that can help you understand how much of your source is tested.

.dockerignore: allows you to mention a list of files and/or directories which you might want to ignore while building the image.

.pylintrc: Pylint is a source-code, bug and quality checker for the Python programming language.

docker-compose.yml: Orchestration file that contains the configuration to develop locally, it overrides the Dockerfile to run the flask development server instead of running the gunicorn server that is used for hosting on Heroku

Dockerfile: File used to create an image to run in a container that runs the program

heroku.yml: The heroku.yml file is a manifest you can use to define your Heroku app. It allows you to Build Docker images on Heroku.

pytest.ini: This is the primary pytest configuration file that allows you to change default behavior.

requirement.txt: Is a type of file that usually stores information about all the libraries, modules, and packages in itself that are used while developing a particular project.

setup.py: is utilized to install the different available packages in Python.
