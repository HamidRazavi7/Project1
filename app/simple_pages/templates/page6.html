To set up the project by using GitHub actions, I created workflows. The workflows run tests on pull or push requests. The development workflow (dev.yml), is responsible for the push requests and the production workflow (prod.yml) is responsible for the push requests. During the push/pull requests, the workflows build the container and push it into the Dockerhub. Consequently, the program is deployed into the Heroku for the automated running.

To review the code using the development server before merging, the following steps are taken:

  1. The master branch where the code is stored is necessary.
  2. Another branch needs to be created and modified before being pushed into the GitHub.
  3. A pull request would trigger the testing.
  4. Once the test is passed, the branch is merged into the master branch.

When pushing the master into the GitHub repo, the code would be tested with pytest. The container is then built and deployed by the production workflow. By creating a production workflow, anytime the code is pushed into the GitHub, a series of actions is triggered that deploys the app into the production server. For this, a DockerHub username and password, as well as a Heroku API is necessary. These would be added under the "secret" tab of the repository setting.

The image of the project can be deployed into the Dockerhub when the master branch is updated. When the master is updated, a pull request would allow merging the master into another branch. This triggers the following steps:

  1. Test on Heroku runtest
  2. Build it on Heroku as a container
  3. Deploy it on Heroku
Once the step 2 is completed, the Docker repository is updated, you can see the image of the code. Anytime a pull request is submitted, it creates an image and saves it on the Dockerhub.
